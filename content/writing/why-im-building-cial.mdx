Cial didn't start as a product vision. It started as a commute problem.

## How I actually used AI

Like a lot of people, my first real taste was **GitHub Copilot** in **late 2021** — not today's ChatGPT-shaped Copilot, the early one (GitHub had just opened the technical preview that summer). Basically a fancy tab accelerator. Sometimes it completed a whole function and you went a bit faster. Already very developer-oriented, but still passive.

![Late 2021 — early GitHub Copilot: ghost text in the editor, Tab to accept](/images/story/early-copilot.svg)

**Late 2022** — I started using **ChatGPT**. I was freelancing and starting **ClinicPilot** with Enzo, and I used it to speed up code iterations. At first, like everyone: copy-paste, generate something from a distance, paste it back. Useful, but still outside the loop.

![Late 2022 — ChatGPT: generate in the browser, paste back into the project](/images/story/chatgpt-copypaste.svg)

Then there was a gap. I got back on track in **mid-2024**, when I moved to **Cursor** — ChatGPT inside the IDE. That's when I stopped being passive.

![Mid-2024 — VS Code → Cursor: the IDE that made chat feel native](/images/story/vscode-to-cursor.svg)

**June 2025** — I fell into the CLI rabbit hole and switched to **Claude Code** (research preview had opened that February; GA was May — so June was still early). I also ran into the OpenClaw idea around then — not that Cial is a clone of it, but the core itch was the same: talk to an agent that actually runs on a machine.

![June 2025 — agent CLIs, Claude Code, the rabbit hole](/images/story/cli-agents.svg)

From there, changing tools became a **muscle habit**. Gemini CLI, different providers, Grok, Mistral — I tried a lot of them, and used many in production. Once you've done the hop once, the next hop is cheaper. That habit is basically the seed of Cial's meta-harness bet later: models and harnesses come and go, the workspace shouldn't have to.

Still, none of that is why Cial exists.

## Three hours a day doing nothing

I work on ClinicPilot every day, out of **Station F in Paris**. I also had **an hour and a half of transport in the morning and the same in the evening**. Three hours where I couldn't really ship.

I wanted to keep developing from my phone. At the time there was no Claude remote control — and even now that it exists, it's not exactly the tool I wanted. So I started building something.

> **Station F, Paris.** An hour and a half each way. Three hours a day on the train, and the agent stayed on the desk.

![First try — Claude Code over Telegram. It worked. It wasn't great. |side](/images/story/telegram-first-try.png)

The first version came from the same place as OpenClaw-style setups: I need to talk to the agent from my phone. I wired it to **Telegram**. It worked. Honestly, it wasn't great. Telegram is fine for chat. It is not built for driving a coding agent on a machine.

## The interface that rewrote itself

So I asked Claude Code to build a proper interface — a small site, frontend + backend, where the backend runs Claude Code, so I could keep talking to it from my phone or any browser. The original idea was basically a **PWA**.

Then the déclic.

> If you have a site the agent is serving… it can redeploy itself. It can modify its own interface live. Rebuild. Ship. Keep going.

That's the moment I thought: ok, there might be something here.

Iteration speed went crazy, because we could iterate on our internal tool *while* using it. In **less than a week**, Cial became the product that basically runs ClinicPilot end to end — development, client stuff, our SaaS operations. Primary tool. And if you strip everything away, it still starts from one idea: **self-edit**.

## Disposable code, continuous product

At some point Enzo and I looked at what we had. We hadn't "bootstrapped a SaaS with Cial" in the fundraising sense. We had pushed ClinicPilot forward *with* Cial as the engine. There was real value there — and not just for us.

By nature, this product almost had to be **open source**.

Because of self-edit, the tech stack of Cial does not matter that much. Today it's common web stuff — Vite, and before that Next, etc. Tomorrow a new language shows up, the day after a new stack. Fine. Throw it out. Rebuild.

Cial is closer to a **direction** than a frozen codebase. The code is relatively disposable. The state at any single point in time is not the important part. **Continuity and evolution** are.

That's what Cial has done since the start: evolve, evolve, evolve.

In under two months we had a platform that is already a serious harness on its own — a **meta-harness**: many models, many harnesses, connect almost anything, do a lot of work well. For the rest, self-edit does the job.

## Why OpenCial

The point of open-sourcing this is simple: let developers try it, break it, tell us what they think, and contribute for *their* use cases. Then, later, make it accessible to non-technical people too — abstract the harness layer, the model juggling, all of it.

Thread by thread, the long game looks like a kind of **liquid UI**: show almost anything, connect almost any backend, drive it in natural language, until model/harness management is invisible.

> Right now it's a web product. Further out — still far — I think this points at a real **OS** direction. Not a claim. A north star.

## What Cial is for, day to day

Today the practical job is continuity.

New models ship every week. New tools every day. People keep switching workspaces and losing the thread. Cial is a thin layer that stays: pull in what you need *now*, implement it, throw away what stopped being useful, keep the sessions, the memory, the shape of how you work.

You don't have to rebuild your life around the model of the month. You stay on Cial, and Cial changes under you.

If that resonates: [cial.ai](https://cial.ai) is the product side, **OpenCial** is the open source, and this site is where I'll keep writing the messy middle — wrong turns included.
